# prnlb-browser Project Instructions

## Architecture

- Organize application code by the five UI features:
  - `src/config/` — configuration persistence, routes, and Config tab client code.
  - `src/crawl/` — crawl orchestration, scraping, routes, and Crawl tab client code.
  - `src/results/` — topic persistence, result routes, export behavior, and Results tab client code.
  - `src/search/` — tracker search, topic-detail fetching, routes, and Search tab client code.
  - `src/downloaded/` — downloaded-item persistence, scanning, preview handling, routes, and Downloaded tab client code.
  - `src/actresses/` — actress catalogue persistence, routes, and Actress tab client code. Has no foreign key to any other table; matching against other tabs' "Cast" text is done by name lookup at the route layer.
- Put functionality used across features in `src/core/`, including Electron bootstrap, preload IPC, HTTP infrastructure, shared types, browser launching, captcha handling, scraping helpers, image resolvers, and shared renderer UI.
- Keep feature-specific code in its owning feature folder. Move code to `src/core/` only when it is genuinely shared or platform-level.
- Do not add new source modules directly under `src/`. Any remaining top-level legacy modules such as `src/server.ts`, `src/db.ts`, `src/scraper.ts`, or `src/search-scraper.ts` are transitional, are not canonical, and must not receive new logic. Migrate references to the feature/core modules instead.

## Backend Conventions

- Register HTTP behavior through the feature `routes.ts` modules composed by `src/core/server/index.ts`.
- Use `AppContext` from `src/core/server/context.ts` for configuration, stores, and shared crawl state instead of introducing new module-level application state.
- Use the HTTP helpers in `src/core/server/http.ts` for JSON bodies/responses, SSE streams, and static files. Do not duplicate request-body or SSE boilerplate in feature routes.
- Preserve existing endpoint paths, response bodies, status codes, and SSE event shapes unless an API change is explicitly requested.
- Keep SQLite access behind the owning feature store (`src/results/store.ts` or `src/downloaded/store.ts`).
- Keep Playwright launching, captcha coordination, image-host resolution, and generic scraping utilities in `src/core/`.

## Renderer Conventions

- Treat each feature's `client.js` as the source for that tab's renderer behavior.
- Put renderer behavior shared by multiple tabs in `src/core/ui/`.
- Do not edit `public/app.js` directly; it is generated by `scripts/build-renderer.js` during `npm run build` (and is gitignored/untracked for that reason). Edits made straight to `public/app.js` build/run without error, so the mistake isn't visible until the next build silently overwrites them from the stale `client.js` — always change the owning feature's `client.js` and regenerate with `node scripts/build-renderer.js` or `npm run build`.
- When adding or moving renderer modules, update the ordered source list in `scripts/build-renderer.js`. The generated scripts are classic scripts sharing one global scope, so source order and globally unique top-level declarations matter.
- Keep `public/index.html` and `public/style.css` as static assets unless a task explicitly changes the frontend build model.

## TypeScript and Imports

- Use strict TypeScript for backend and Electron code.
- The project uses `module: "nodenext"` with CommonJS output. Use explicit `.js` extensions in relative TypeScript imports, for example `import { json } from "../core/server/http.js"`.
- Prefer `node:` specifiers for Node built-ins.
- Keep shared domain interfaces in `src/core/types.ts`; keep feature-only interfaces with their feature.
- Avoid `any`; narrow `unknown` values and use explicit request-body types.

## Build and Test

- Run `npm run build` after source changes. It cleans `dist/`, regenerates `public/app.js`, and compiles TypeScript.
- Run `npm test` after behavior changes. Tests use the Node test runner through `tsx`, so `better-sqlite3` must be built for the Node ABI first; if `npm test` fails with `NODE_MODULE_VERSION` mismatches, run `npm rebuild better-sqlite3`, then `npm run rebuild` to restore the Electron ABI before launching the app.
- Add focused tests under `test/` for extracted helpers, stores, parsers, and route infrastructure. Tests use the Node test runner through `tsx`.
- For Electron or packaging changes, also verify `npm run electron`; use the relevant `npm run dist:<platform>` command when packaging behavior changes.
- Never commit credentials, passwords, local database paths, downloaded-folder paths, or other machine-specific values from `config.json`. The bundled default configuration is `config.template.json` (credentials-free); the live `config.json` is created inside Electron's user data directory at runtime.

## Change Discipline

- Make feature moves atomically: update imports, tests, Electron entry paths, and renderer build inputs in the same change.
- Avoid compatibility shims unless they are required for a staged migration; remove them once all imports use the canonical path.
- Preserve the local-only data model and Electron security settings (`contextIsolation: true`, `nodeIntegration: false`). Expose native operations only through the minimal preload API.
- Update `README.md` when build commands or the source layout change.
